# Study Scan Skill

자료를 깊게 다룰 가치가 있는지 빠르게 판단한다.

## 사용법

```text
/study-scan <source_url_or_artifact>
```

## 빠른 판별 순서

### 0. 출처 사슬 확인

먼저 지금 받은 링크가 원자료인지 포인터인지 구분한다.

| 링크 유형 | 처리 |
|---|---|
| 논문, 저장소, 릴리스, 공식 문서 | 원자료로 본다 |
| LinkedIn, Threads, 블로그 공유글 | 안에 연결된 논문, 저장소, 제품, 문서가 있는지 찾는다 |
| 제품 URL | 제품 자체를 원자료로 보고 화면, 문서, API, 코드 공개 여부를 확인한다 |
| 요약글 | 요약글의 근거가 되는 원자료를 먼저 찾는다 |

원자료가 확인되면 scan의 기준은 공유글이 아니라 원자료다. 공유글은 맥락, 작성 의도, 연결 경로로만 기록한다.

### 1. 30초 확인

다음만 빠르게 본다.

- 제목
- 요약, abstract, 릴리스 제목, README 첫 부분
- 섹션 제목
- 표, 그림, changelog bullet
- 저장소 metadata
- 연결된 원자료 링크

### 2. 판별 질문

YES, NO, MAYBE로 답한다.

| 질문 | 답 | 메모 |
|---|---|---|
| AI agent, 자동화, 소프트웨어 운영, 회계, 세무, 규정 업무와 관련되는가 | | |
| 새롭거나 유용한 패턴이 있는가 | | |
| 코드, 논문, 문서, 릴리스 근거가 있는가 | | |
| 작성자, venue, 프로젝트, artifact가 학습할 만큼 신뢰 가능한가 | | |
| 현재 시스템이나 미래 실무에 도움이 되는가 | | |
| 공유글보다 더 우선해야 할 원자료가 있는가 | | |

### 3. 분류

| 분류 | 뜻 |
|---|---|
| 반드시 다룸 | 관련성, 새로움, 적용성이 모두 높음 |
| 다룰 만함 | 유용한 패턴이나 참고 자료 |
| 훑기 | 배경지식 수준 |
| 보류 | 관련성이나 근거가 부족함 |

### 4. 우선순위 점수

```text
관련성 (0-4):
새로움 (0-3):
적용성 (0-3):
합계:
```

## 출력 형식

```markdown
# 빠른 판별: [제목]

**판정**: [반드시 다룸 / 다룰 만함 / 훑기 / 보류]
**우선순위**: X/10
**예상 시간**: [예상 소요]

## 30초 요약
[두 문장 이내]

## 관련성
[왜 중요한지 한 문장]

## 근거
- 출처:
- 코드/문서:
- 릴리스/논문:

## 결정
[지금 다룸 / 나중에 큐에 넣음 / 보류]

## 볼 지점
- 항목:
- 이유:
 - 원자료 우선순위:
```
