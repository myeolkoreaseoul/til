# Study Analysis Skill

공개 기술 자료를 깊게 분석한다.

## 사용법

```text
/study-analysis <source_url_or_artifact>
```

## 6단계 분석 틀

### 1. 출처 사슬과 Metadata 추출

- **제목**
- **작성자 또는 소유자**
- **자료 종류**
- **게시일 또는 갱신일**
- **링크**: 출처, 저장소, 릴리스, 문서, 논문, demo, package
- **키워드**
- **원자료 우선순위**: 분석 기준이 되는 원자료를 먼저 쓰고, 공유글은 보조 맥락으로 내린다.

| 상황 | 분석 기준 |
|---|---|
| LinkedIn 글이 저장소를 소개함 | 저장소 README, docs, code, release를 기준으로 분석 |
| LinkedIn 글이 논문을 소개함 | 논문 PDF, official project page, code를 기준으로 분석 |
| LinkedIn 글이 제품을 소개함 | 제품 화면, docs, pricing, API, 공개 코드 여부를 기준으로 분석 |
| 작성자 의견 글 | 의견 자체를 분석하되 근거 링크가 있으면 따로 확인 |

### 2. 문제 정의

다음 질문에 답한다.

| 질문 | 쓸 내용 |
|---|---|
| 핵심 문제 | 무엇을 해결하는가 |
| 동기 | 왜 지금 중요한가 |
| 기존 한계 | 기존 방식이 무엇을 못 했는가 |
| 학습 질문 | 이 자료에서 무엇을 배우려 하는가 |

### 3. 방법 또는 구조

| 층 | 쓸 내용 |
|---|---|
| 직관 | 전문 용어 없이 핵심을 설명 |
| 기술 접근 | 구조, workflow, algorithm, contract, 구현 패턴 |
| 구현 메모 | 명령어, 설정, runtime 경계, dependency, 확인 지점 |

제품이나 저장소가 원자료이면 다음을 추가로 확인한다.

| 층 | 확인할 것 |
|---|---|
| 화면 | 사용자가 실제로 보는 화면, 입력, 출력 |
| 서버/API | endpoint, data shape, cache, auth, rate limit 대응 |
| 데이터 | 실데이터, 시연 데이터, fixture, 외부 API 경계 |
| 핵심 로직 | 알고리즘, scoring, rule, workflow engine |
| 설정 | env, credential, deploy, 실행 명령 |
| 검증 | test, backtest, benchmark, 직접 실행 결과 |
| 한계 | 원문이 밝힌 제한, 데이터 품질, 미검증 주장 |

### 4. 근거와 결과

| 구성 | 확인할 것 |
|---|---|
| 출처 근거 | README, release note, docs, code, examples |
| 변경 또는 기능 | 무엇이 바뀌었거나 존재하는가 |
| 주장 | 원자료가 무엇을 주장하는가 |
| 검증 경로 | 어떻게 확인하거나 실험할 수 있는가 |
| 한계 | 원자료가 경고하거나 부족한 부분 |

### 5. 비판적 분석

| 항목 | 질문 |
|---|---|
| 강점 | 무엇을 잘하는가 |
| 약점 | 무엇이 부족하거나 위험한가 |
| 분석 관점 | 기법, 구성요소, 운영 적용성, 연결 지점 |
| 질문 | 더 확인할 것은 무엇인가 |

### 6. 실행 가능한 정리

- 짧은 원문 근거
- 관련 이슈
- 적용 가능 요소
- 보류할 요소
- 다음 행동

## 품질 체크리스트

- [ ] 6단계가 모두 들어갔다.
- [ ] 출처 URL이 들어갔다.
- [ ] 공유글과 원자료를 구분했다.
- [ ] 제품 또는 저장소라면 화면, API, 핵심 로직, 데이터 경계, 실행 확인을 봤다.
- [ ] 공개 이슈에 내부 점검 결과를 쓰지 않았다.
- [ ] 적용 가능 요소가 구체적이다.
- [ ] 관련 이슈가 있으면 연결했다.
